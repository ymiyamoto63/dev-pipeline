**言語: すべての出力文書、コミットメッセージ、GitHub Issueの内容は日本語で書くこと。**

**再開モード:** {{claude:引数}}{{copilot:メッセージ}}が`resume #<issue番号>`である場合（または明らかに既存の実行を指している場合）、`docs/<issue番号>/pipeline-state.md`を読み、そこに記載されたブランチにまだいなければチェックアウトし、記録されているリトライ回数を保ったまま最初の未完了フェーズから続行すること。`docs/<issue番号>/`内の既存文書は完了済みフェーズの正式な記録である — やり直さないこと。

**要件定義済み入力:** {{claude:引数}}{{copilot:メッセージ}}が既存の要件定義書へのパス（あるいは明らかにそれを指すもの） — 典型的にはrefine-requirementsスキルが作成した`docs/requirements/<feature-slug>.md` — である場合、フェーズ1は新規文書を作成する代わりにその文書から開始する（フェーズ1の*事前に要件定義済みの入力*の注記を参照）。それ以外はすべて変わらず実行される: 承認ゲート、GitHub Issue作成、`docs/<issue番号>/`ディレクトリ、featureブランチ、stateファイル、チェックポイントコミットはすべて通常実行と全く同じように行われる。事前に書かれた要件定義書はアナリストによる文書作成作業を省略する理由にはなるが、Issue作成やそれ以降のいかなるステップも省略する理由には決してならない。

**プロジェクトのpipeline config:** フェーズ1の前に、`<project_root>/docs/pipeline-config.md`が存在するか確認すること。存在する場合、それはプロジェクトのスタック概要とフェーズごとのルールを含んでいる: その`## stack`節をすべてのサブエージェントへのプロンプトに含め、さらに各サブエージェントにはそのフェーズ専用の節（software-architectには`## design`、code-reviewerには`## review`、というように）も渡すこと。存在しない場合は、リポジトリ（package.json、pom.xml / build.gradle、devcontainer設定、…）から自分でスタックを検出し、すべてのサブエージェントに1行程度のスタック概要を渡し、実行の最後にユーザーへ`docs/pipeline-config.md`の追加を提案すること（テンプレート: dev-pipelineリポジトリの`templates/pipeline-config.md`）。

**検証済みコマンド:** ビルド/型チェック/lint/テストのコマンドは実行ごとに一度だけ確定し、以降は再利用する — サブエージェントにフェーズごとの再発見を決してさせないこと。configに`## commands`節があればそれが確定値である。なければ、遅くともフェーズ2の終わりまでに自分でリポジトリから一度だけ発見し（package.jsonのスクリプト、mvnw/gradlewラッパー — 決して推測しない）、`pipeline-state.md`の`Commands`行に記録すること。ビルドやテストを行うすべてのサブエージェント（implementer、test-engineer）には確定値として渡すこと: 彼らはそのコマンドをそのまま実行し、動かない場合や必要なツールが不足している場合は報告する — 何かをインストールしたり代替手段を探し回ったりはしない。

**プリフライト:** フェーズ1の前に`git status`を実行すること。作業ツリーが汚れている場合は停止し、{{claude:AskUserQuestion経由で}}{{copilot:チャットで直接}}ユーザーに、先にクリーンアップするために中断するか、既存の変更に一切手を触れずに続行するかを確認すること — 見慣れない変更がこのタスクのものだと決して仮定しないこと。現在のブランチと`git rev-parse HEAD`を控えておくこと。

**ブランチとチェックポイントコミット:** パイプラインは専用ブランチ`feature/<issue番号>-<short-slug>`上で作業する。このブランチはフェーズ1の最後に作成され、作成元のHEADが**ベースSHA**となる — 以降のすべての差分は`ベースSHA..HEAD`である。進行に合わせてチェックポイントをコミットすること: フェーズ1の後（要件定義書 + stateファイル）、フェーズ2の後（設計書）、ビルドチェックを通過した実装ステップごとの後、そしてリトライループでの修正ごとの後 — 関与したファイルを名前でステージし（`git add -A`は絶対に使わない）、各チェックポイントコミットに`docs/<issue番号>/pipeline-state.md`を含めること。メッセージの形式: `<フェーズ/ステップの要約> (#<issue番号>)`、日本語で。チェックポイントがあることで、各リトライは既知の正常な地点に戻ることができ、レビュー用の差分がパイプラインの作業内容とちょうど一致する。

**パイプラインの状態:** issue番号が判明したらすぐに`docs/<issue番号>/pipeline-state.md`を作成し、フェーズごと・リトライの繰り返しごとに更新すること — このファイルがあることで、中断した実行を再開できる。形式:

```
# Pipeline State
- Issue: #123
- Branch: feature/123-avatar-upload
- Base SHA: abc1234
- Mode: confirm-design | auto
- Commands: pnpm typecheck / pnpm test / ./mvnw test
- [x] 1. requirements（承認済み）
- [x] 2. design
- [ ] 3. implementation（step 2/4）
- [ ] 4. testing（retry 0/3）
- [ ] 5. review（retry 0/2）
- [ ] 6. publish
```

**文書ディレクトリ:** 各フェーズのサブエージェントは、出力文書を`<project_root>/docs/<issue番号>/`（`<issue番号>`はフェーズ1で採番されたGitHub Issue番号）に書き込む。各サブエージェントには正確なdocsディレクトリのパスを明示的に伝えること。後のフェーズに委譲する際は、前のフェーズの内容全文をプロンプトに貼り付けるのではなく、その文書パスを指し示すこと — サブエージェントには読み取り権限があり、ファイルが正である。

**委譲のボイラープレート:** すべての委譲には次を含めること: プロジェクトルート、正確なdocsディレクトリのパス、スタック概要（configの`## stack`節、なければ自作の1行）、検証済みコマンド（ビルドやテストを行うフェーズ向け）、そのフェーズ専用のconfig節（configが存在する場合）、そのフェーズのlessons抜粋（あれば）。以下のフェーズ一覧ではこれを繰り返さない — 常に適用される。

**Lessons learned（失敗の記憶）。** `<project_root>/docs/lessons-learned.md`は、上記のフェーズごとの文書とは別の累積ログである: 上書きされることはなく、追記のみで、パイプラインの実行をまたいで残り続けるため、過去の失敗が今後に活かされる。フェーズごとに1つの節として整理され、各エントリには日付が付く:

```
# Lessons Learned

## <phase>            … requirements / design / implementation / testing / review / publish

### YYYY-MM-DD — <short title>
- **失敗**: 具体的に何が壊れたか
- **根本原因**: なぜ発生したか
- **修正**: 何が解決したか（リトライ予算が尽きた場合は「未解決 — パイプライン停止」）
- **予防策**: 次回同フェーズのサブエージェントへの具体的・実行可能な指針
```

フェーズ1の前に一度（存在すれば）読み、委譲する際は各サブエージェントのプロンプトに、その自身のフェーズの節からのエントリのみを抜粋すること — すべてのフェーズに全文を読ませないこと。履歴が長くなると、すべての呼び出しが肥大化してしまう。（既存ファイルがまだ旧来のフラットな形式 — `## date — phase — title`という見出し — を使っている場合は、全文を読み、各見出しに書かれたフェーズで抜粋すること。）

以下のフェーズ4、5、6のいずれかでリトライループが発生した場合（テスト失敗、ブロッキングなレビュー指摘、公開のブロッカー）、それが解決したら — あるいはリトライ予算が尽きて停止したら — 該当するフェーズの節の下に1エントリを追記すること（必要であれば`# Lessons Learned`見出し、あるいは不足している節ごと新規作成すること）。追記する前に、その節に既に同じ根本原因を説明する既存エントリがないか確認すること: あれば、重複させず、そのエントリを更新すること（新しい日付を追加し、予防策を研ぎ澄ます）。エントリは短く具体的に保つこと — このファイルはスキャンしやすい状態を保ってこそ役に立つ。ルーチンで再発しない問題（例: リトライループなしで実装者が自力で直した単発のタイポ）は記録しないこと。ファイルのエントリ数が約30件を超えたら、実行の最後にユーザーへ統合を提案すること。

フェーズは以下の順序:

1. **要件定義（Requirements）** — 生のタスク説明とプロジェクトルートを添えて{{claude:subagent_type `requirements-analyst`を呼び出す}}{{copilot:`requirements-analyst`エージェントをサブエージェントとして呼び出す}}。`docs/requirements.md`（issue番号が判明する前の一時的な置き場所）に書き込まれる。

   *事前に要件定義済みの入力:* タスクが既存の要件定義書である場合（上記参照）、文書作成の呼び出しはスキップすること — 既に設計に着手できる状態の文書をアナリストに書き直させないこと。文書は自分で読み、その「未決定事項」節を以下のインタビューループにおける未決定事項として扱うこと: スコープや設計に実質的な影響を与える項目をユーザーと一緒に解決すること。各項目の「推奨仮置き」を、記録されている根拠とともに推奨選択肢として提示し、その後`requirements-analyst`を更新モードで再度呼び出して（文書パスと回答を渡す）文書に反映させること。ユーザーが明示的に保留した項目は未決定事項として残すこと。文書に実質的な未決定事項がなければ、そのまま承認ゲートに進むこと — 承認ゲート以降のすべて（Issue、`docs/<issue番号>/`、ブランチ、stateファイル、チェックポイントコミット）は、このモードでも決して省略されない。

   *インタビューループ:* スコープや設計に実質的な影響を与える未決定事項が報告された場合、{{claude:AskUserQuestion経由でユーザーと}}{{copilot:チャットで直接ユーザーに尋ねて回答を待つことで}}それらを解決すること（スコープを変える事項について推測しないこと）。アナリストは各質問に対して根拠付きの推奨回答を用意している — {{claude:それを最初の選択肢とし、ラベル末尾に「（推奨）」を付け、説明欄に根拠を記載すること}}{{copilot:それを最初に提示し、「（推奨）」の印と根拠を添えること}}。その後に本当の代替案を続けること。これにより、ユーザーは{{claude:素早く}}{{copilot:「すべて推奨どおり」といった短い返信で}}デフォルトを受け入れられる。推奨はあくまで提示すべき提案であり、既に下された決定ではない — 推奨があるからといって質問をスキップしないこと。浅い一巡で済ませず、曖昧さを徹底的に掘り下げること — 本当の未決定事項が残っている限り質問を続け、ループ全体を通じてユーザーに尋ねる質問数は最大20問までとする（個々の質問数をカウントし、{{claude:AskUserQuestionの呼び出し数}}{{copilot:チャットのやり取り数}}やラウンド数ではない）。回答はラウンドをまたいでためておき、その後`requirements-analyst`を更新モードで**一度だけ**呼び出すこと — 文書パスとすべての回答をまとめて渡し、文書自体に反映させること（自分で文書を編集しないこと。アナリストが構造の一貫性を保つ）。ループ途中での追加の更新呼び出しは、回答によって状況が大きく変わり、次の質問を組み立てるのに新たなリポジトリ調査が必要になる場合に限り正当化される。上限に達してもなお曖昧さが残る場合は、未解決の点をユーザーに率直に伝え、推測せずどう進めるかをユーザーに決めてもらうこと。

   *承認ゲート:* 未決定事項が解決したら、ユーザーにコンパクトな要約 — スコープ、受け入れ基準（手動確認タグの付いたものにはフラグを立てる）、非目的 — を見せ、{{claude:1回のAskUserQuestion呼び出しで}}{{copilot:チャットで直接ユーザーに尋ね、返信を待って}}: (a) 要件を承認するか変更を要求するか、(b) 実行モードを選ぶ — **confirm-design**（設計承認の際にもう一度立ち止まる）または**auto**（フェーズ2〜5をノンストップで実行。公開は常にモードに関わらず確認が必要）。承認前にGitHub Issueを作成しないこと。

   承認後: `gh issue create`でGitHub Issueを作成する — タイトル: タスク要約を日本語で（簡潔に、1行）。本文: `--body-file docs/requirements.md`で要件定義書そのものを渡す（文書の内容をコマンドラインに貼り付けたり、投稿のためだけにファイルを読んだりしないこと）。`requirements`ラベルが存在すればそれを付け、なければ何も付けない。表示されたURLからissue番号を取得すること。`docs/requirements.md`を`docs/<issue番号>/requirements.md`に移動する（先にディレクトリを作成する） — あるいは、事前に要件定義済みの入力の場合は、代わりに元の文書を`docs/<issue番号>/requirements.md`に**コピー**し、原本は`docs/requirements/`配下にそのまま残すこと。以降、`docs/<issue番号>/`内のコピーがこの実行における正の文書となる。それ以降のすべての文書はそこに置かれる。現在のHEADから`feature/<issue番号>-<short-slug>`ブランチを作成し、そのHEADをベースSHAとして記録すること。`pipeline-state.md`を書き、最初のチェックポイントコミット（要件定義書 + stateファイル）を行うこと。

2. **設計（Design）** — `docs/<issue番号>/requirements.md`を指し示して{{claude:subagent_type `software-architect`を呼び出す}}{{copilot:`software-architect`エージェントをサブエージェントとして呼び出す}}。`docs/<issue番号>/design.md`に書き込まれる。先に進む前に設計書を確認すること: 要件定義書のすべての受け入れ基準をカバーするACマッピング節が含まれていること。タスクがフロントエンド/バックエンド境界をまたぐ場合はAPIコントラクト節（エンドポイント、リクエスト/レスポンスの形状、エラーレスポンス）が含まれていること。加えてconfigの設計ルールが追加で要求する事項（例: スキーマ変更時のDBマイグレーション計画）。何か欠けている場合は、その欠落を埋めるため一度だけ`software-architect`に差し戻すこと。

   設計内容をIssueに投稿すること: `## 設計`見出しに`docs/<issue番号>/design.md`を続けた一時ファイルをシェルの連結で組み立て、`gh issue comment <issue番号> --body-file <ファイル>`で投稿する — 文書の内容をコマンドラインに貼り付けたり、投稿のためだけにファイルを読んだりしないこと。その後、ユーザーに短い要約（影響を受けるファイル+アプローチ、数文）を見せること。**confirm-design**モードでは、続行前に{{claude:AskUserQuestion経由で}}{{copilot:チャットで}}承認を待つこと。**auto**モードでは即座に続行すること — この要約はゲートではなく状況報告である。設計書 + stateファイルをチェックポイントコミットすること。

3. **実装（Implementation）** — 設計の実装ステップごとに1回（またはきつく関連する小さなステップをまとめて）、`docs/<issue番号>/requirements.md`と`docs/<issue番号>/design.md`の該当部分、加えて正確なファイルパスを指し示して{{claude:subagent_type `implementer`を呼び出す}}{{copilot:`implementer`エージェントをサブエージェントとして呼び出す}}。設計のステップ順に従うこと — 境界をまたぐ変更では典型的にDBマイグレーション→バックエンド→フロントエンドの順になり、各ステップがコンパイル可能な状態の上に積み上がる。各呼び出しは`docs/<issue番号>/implementation-notes.md`に追記していく。各ステップの検証が通過したら、そのステップのファイル（+ notes + stateファイル）をチェックポイントコミットすること。メッセージは`step <n>/<total>: <ステップ名> (#<issue番号>)`のような形。最後のステップの後、`git diff <base SHA> --stat`で、ブランチが実際に設計の「影響を受けるファイル」リストと一致する変更を含んでいることを、テストフェーズに進む前に確認すること。

4. **テスト（Testing）** — `docs/<issue番号>/requirements.md`、`docs/<issue番号>/design.md`（そのテスト戦略節）、`docs/<issue番号>/implementation-notes.md`を指し示して{{claude:subagent_type `test-engineer`を呼び出す}}{{copilot:`test-engineer`エージェントをサブエージェントとして呼び出す}}。どちら側が変更されたかを伝えること — 変更されたすべての側（フロントエンドおよび/またはバックエンド）のテストスイートを実行しなければならず、片方だけではいけない。`docs/<issue番号>/test-report.md`に書き込まれる。
   - 失敗が報告された場合: 新しい`implementer`{{claude:呼び出し}}{{copilot:の呼び出し}}に差し戻して修正させること — 言い換えず、`docs/<issue番号>/test-report.md`（正確なコマンドとエラー出力）を指し示すこと — 修正をチェックポイントコミットし、`test-engineer`を**失敗にスコープを絞って**再実行すること: どのテスト/スイートが失敗したかを正確に伝え、まずそれらを再実行させ、通過した場合に限り同じ呼び出しの中で関連するフルスイートに進ませる — リトライのたびにフルスイート実行のコストを払わないこと。最大3回繰り返す。それでも失敗する場合は、ループを続けずに停止し、ユーザーにブロッカーを報告すること。いずれの場合も（修正できた場合、予算が尽きた場合のいずれも）、上記の形式でlessonエントリを追記すること。

5. **レビュー（Review）** — `docs/<issue番号>/requirements.md`、`docs/<issue番号>/design.md`、`docs/<issue番号>/implementation-notes.md`、`docs/<issue番号>/test-report.md`を指し示し、ベースSHAを渡して{{claude:subagent_type `code-reviewer`を呼び出す}}{{copilot:`code-reviewer`エージェントをサブエージェントとして呼び出す}}。実装はチェックポイントコミットとして存在するため、`git diff <base SHA>..HEAD`をレビューし、テストレポートから何が既に検証済みかを把握する。`docs/<issue番号>/review.md`に書き込まれる。
   - スタイルではなく正確性レベルの指摘がある場合: `implementer`に差し戻して修正させ、修正をチェックポイントコミットし、`test-engineer`（フェーズ4と同様、修正が触れた範囲にスコープを絞る）と`code-reviewer`を**再レビューモード**で実行すること: レビュアーには前回の指摘と修正コミットの範囲を渡し、各指摘が解消されたことの確認と修正差分そのもののレビューを行わせる — ブランチ全体の再レビューはさせない。最大2回繰り返す。問題が解消しない場合は、公開せずに停止し、未解決の指摘をユーザーに要約すること。いずれの場合も、上記の形式でlessonエントリを追記すること。

6. **公開（Publish）** — autoモードであっても、常に{{claude:まず}}{{copilot:まずチャットで、返信を待って}}明示的な確認をユーザーに求めること（これはリモートへのpushとPRのオープンという、目に見える、後戻りしにくい操作であるため）。確認が得られたら、`docs/<issue番号>/requirements.md`、`docs/<issue番号>/design.md`、`docs/<issue番号>/test-report.md`を指し示し、ベースSHAとissue番号を渡して{{claude:subagent_type `pr-publisher`を呼び出す}}{{copilot:`pr-publisher`エージェントをサブエージェントとして呼び出す}}。残っている未コミットのファイル（通常は`test-report.md`、`review.md`、stateファイル）をコミットし、`docs/<issue番号>/pr-description.md`を保存し、pushして、Issueがマージ時に自動的にクローズされるよう本文に`Closes #<issue番号>`を入れてPRを開く。
   - PR作成後: Issueに（`gh issue comment`で）テスト結果の一文要約とPR URLを添えたクロージングコメントを投稿し、`pipeline-state.md`のフェーズ6を完了にマークし、featureブランチ上でその最終的なstate更新をコミット・pushすること。
   - `pr-publisher`が公開せずに停止した場合（`gh`未認証、リモートなし、秘密情報の疑い、誤ったブランチ）、次回の実行が同じブロッカーに引っかからないよう、上記の形式でlessonエントリを追記すること。

全体を通して: 各サブエージェントの生の出力全文を垂れ流すのではなく、フェーズ間でユーザーに1行の状況報告（何が完了し、次は何で、どの文書を書いたか）を伝えること。最後に、PR URL（またはパイプラインが停止した場所とその理由）を報告し、全体の記録は`docs/<issue番号>/`にある（この実行で追記があれば`docs/lessons-learned.md`も）こと、中断した実行は`/dev-pipeline resume #<issue番号>`で再開できることをユーザーに伝えること。
